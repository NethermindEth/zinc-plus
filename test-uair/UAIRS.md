# UAIRs in `zinc-test-uair`

This crate defines the UAIRs (Univariate AIR-style constraint systems) that
the prover, verifier, PIOP, and end-to-end benchmarks exercise. Each UAIR is
a self-contained example: a column layout, optional row shifts, a constraint
function, and a random-trace generator. The constraints themselves are
written against the `ConstraintBuilder` DSL — the prover and verifier each
provide their own builder implementations, so the same constraint code is
reused for both witness generation and verification.

This document is a reference for every UAIR currently in
[src/lib.rs](src/lib.rs): what its shape looks like, what it constrains,
what its trace generator produces, and what part of the system it exists to
exercise.

## Conventions

### Column kinds

A `TotalColumnLayout::new(b, a, i)` declares `b + a + i` columns of three
distinct kinds. Each column kind stores a different value type per row:

- **Binary-polynomial columns** (`bp`) hold `BinaryPoly<32>` — a polynomial
  with 32 coefficients, each restricted to `0` or `1`. Note that the
  *coefficients* are binary; the column entry itself is still a polynomial,
  not a single bit. Useful identities: `bp.eval(1)` is the popcount (number
  of `1` coefficients) and `bp.eval(2)` is the integer encoded by reading
  the coefficients as the bits of a `u32`.
- **Arbitrary-polynomial columns** (`ap`) hold `DensePolynomial<R, 32>` — a
  polynomial of degree `< 32` with arbitrary coefficients in the host ring
  `R`.
- **Integer columns** (`int`) hold a single `R` element per row (a scalar,
  with no polynomial structure).

Throughout this document, `bp[i]`, `ap[i]`, and `int[i]` refer to the
`i`-th column of each kind.

### Rows: `up` and `down`

The constraint function receives two row views per call:

- `up.X[i]` is the value of column `X[i]` at the current row `t`.
- `down.X[i]` is the value of column `X[i]` at the *shifted* row.

**Important — `down` is indexed by shift-spec position, not by source
column.** The signature declares a `Vec<ShiftSpec>`; the `j`-th entry of
`down.binary_poly` corresponds to the `j`-th `ShiftSpec` in that vector,
not to `bp[j]`. Concrete examples:

- `BigLinearUair` uses `(0..16).map(|i| ShiftSpec::new(i, 1))`. The shift
  vector is in source-column order, so `down.binary_poly[i]` happens to
  equal `bp[i][t+1]` here.
- `SHAProxy` uses 21 shift specs (17 binary-poly + 4 int) with shift
  amounts of 9, 13, 16, and 17 — including **multiple shift amounts on
  the same source column** (`bp[0]` and `bp[1]` each get three shifts;
  `bp[4]` gets two). This is the only UAIR in the codebase that mixes
  binary-poly and int shifts in the same shift vec, and the only one
  where the same `source_col` appears with multiple amounts.
  `UairSignature::new` automatically sorts the shift vec stably by
  `source_col`, so writing the shift specs in source-column order in the
  signature makes the `down.binary_poly[k]` indexing in the constraint
  code mechanical to derive.

Shifts are zero-padded beyond the trace boundary: the "virtual" row past
the last real row reads as all zeros.

### Constraints

A `ConstraintBuilder` exposes two assertion primitives:

- `b.assert_zero(expr)` — the expression must reduce to the zero element.
- `b.assert_in_ideal(expr, &ideal)` — the expression must lie in the given
  ideal of `R[X]`. At verification time, expressions are projected to a
  prime field and the membership check is evaluated there.

Expressions are built from column values using `+`, `-`, `*`, plus two
extras supplied to the constraint function:

- `from_ref(&scalar)` lifts a fixed `Self::Scalar` (a
  `DensePolynomial<R, 32>`) into the expression type — i.e., a constant
  polynomial term.
- `mbs(&expr, &scalar)` multiplies an existing expression by a fixed
  `Self::Scalar`. This is the only way to multiply a column expression by
  a polynomial constant such as `X` (encoded as `DensePolynomial::new([0, 1])`).

### Ideal types

Two ideal implementations ship today:

- **`DegreeOneIdeal<R>`** generated by `X - r` for some root `r ∈ R`. An
  expression `e(X)` lies in `(X - r)` iff `e(r) = 0` (i.e., the projected
  polynomial vanishes at `r` in the field). This document writes such
  ideals as `(X - r)`. The two roots used in practice are `r = 1` (so
  `(X - 1)` checks "popcount/sum" relations) and `r = 2` (so `(X - 2)`
  checks "integer-decomposition" relations).
- **`XnMinusOneIdeal<N>`** generated by `X^N - 1`. An expression
  `e(X) ∈ R[X]` lies in `(X^N - 1)` iff its coefficients fold to zero
  modulo `N`: for each `k ∈ 0..N`, `Σ_{i ≡ k (mod N)} e_i = 0` in the
  field. Geometrically, `R[X]/(X^N - 1)` is the ring where multiplying
  by `X^k` is a cyclic left-shift of the coefficient vector by `k`,
  which makes this ideal a natural fit for cyclic-bit-shuffle relations
  (e.g. SHA-style σ/Σ functions). Currently used only by `SHAProxy`
  with `N = 32`.
- **`MixedDegreeOneOrXnMinusOne<R, N>`** is an enum that wraps either
  a `DegreeOneIdeal<R>` or an `XnMinusOneIdeal<N>`. UAIRs that need
  both kinds of ideals in the same constraint system use this as
  `Self::Ideal`. Currently used only by `SHAProxy` with `N = 32`.
- **`ImpossibleIdeal`** has `contains(_) = false`. It is the placeholder
  type used by UAIRs whose constraints are exclusively `assert_zero`
  (no ideal-membership check is ever performed against this type).

## Benchmark UAIRs

These UAIRs are sized for the end-to-end criterion benchmark in
[protocol/benches/e2e.rs](../protocol/benches/e2e.rs). They are deliberately
"wide" (many columns and many constraints) so that prover/verifier
performance numbers reflect realistic per-constraint and per-column
overheads rather than fixed-cost setup.

### `BigLinearUair<R>`

A many-column linear UAIR encoding a *popcount-preservation* relation:
each binary-polynomial column is allowed to permute its `1` bits across
rows, but its popcount must stay constant; in addition, column 0 is
forced to be the `BinaryPoly` representation of the row's total popcount
across all 16 columns.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Shifts:** all 16 bp columns shifted by 1 (so `down.binary_poly[i] =
  bp[i][t+1]`, taking advantage of the source-column order in the shift
  vector)
- **Ideal:** `(X - 1)` and `(X - 2)`
- **Constraints (per row `t`):**
  ```
  (bp[0] + bp[1] + … + bp[15]) - int[0]            ∈ (X - 1)   // int[0] = total popcount
  down.binary_poly[0]          - int[0]            ∈ (X - 2)   // bp[0][t+1] encodes that popcount as a u32
  for i in 1..16:
      up.binary_poly[i]        - down.binary_poly[i] ∈ (X - 1) // bp[i] preserves popcount across rows
  ```
- **Trace:** row 0 is fully random; for each subsequent row, `bp[1..16]`
  are fresh random binary polynomials that *match the previous row's
  popcount on each column*, and `bp[0][t+1]` is set to
  `BinaryPoly::from(total_popcount)` so that the second constraint holds.
- **Why it exists:** stresses the prover's per-column and per-constraint
  cost (16 columns × 17 constraints, all degree 1) without introducing any
  non-linearity. The matching MLE-first prover path also runs because the
  max constraint degree is 1.

### `BigLinearUairWithPublicInput<R>`

Identical to `BigLinearUair` (same layout, shifts, constraints, trace),
but the first 4 binary-polynomial columns are marked as public input.
Exists so the end-to-end benchmark can also measure the public-input
plumbing on the same constraint shape.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Public layout:** `(4 bp, 0 ap, 0 int)` — first 4 bp columns are public
- **Shifts / ideal / constraints / trace:** delegated to `BigLinearUair`

### `SHAProxy<R>`

A "wide and shifty" linear UAIR designed to stress the protocol's
shift-handling, `mbs` machinery, and **mixed-ideal** support. It has
14 binary-polynomial columns, 4 integer columns, **21 shift specs**
(17 bp + 4 int) with **multiple shift amounts on the same source
column**, **scalar polynomial multiplications** with multi-term,
high-degree scalars (`X^25 + X^14` and `X^25 + X^13`), and uses both
`(X - 2)` and `(X^32 - 1)` ideal types in the same UAIR.

The name reflects the inspiration — the constraint shape is reminiscent
of bit/integer relations found in proxies for SHA-style hash circuits
(specifically, the cyclic-shift-and-mix structure of SHA's σ/Σ
functions) — even though SHAProxy does not actually compute a SHA
round.

The user-facing constraints are written in a `t`-relative frame with
**negative shifts** (`bp[k][t-N]`). Since `ShiftSpec` only supports
forward shifts, the constraint set is **globally re-indexed** by
substituting `s = t - 16` so that every reference becomes a forward
shift in `s`. The constraint *shape* (which columns interact, with what
coefficients) is preserved exactly. Maximum forward shift in the
`s`-frame is 17.

- **Layout:** `(14 bp, 0 ap, 4 int)`
- **Shifts:** 21 specs total (17 binary-poly + 4 int), with shift
  amounts of 9, 13, 16, and 17. Multiple shifts per source column:
  - `bp[0]`: shifts 13, 16, 17 → `down.binary_poly[0..3]`
  - `bp[1]`: shifts 13, 16, 17 → `down.binary_poly[3..6]`
  - `bp[2]`, `bp[3]`: shift 16 → `down.binary_poly[6..8]`
  - `bp[4]`: shifts 9, 16 → `down.binary_poly[8..10]`
  - `bp[5..=11]`: shift 16 → `down.binary_poly[10..17]`
  - `int[0..=3]`: shift 16 → `down.int[0..4]`
  - `bp[12]` and `bp[13]` are unused fillers (no shift, no constraint).
- **Ideal:** `MixedDegreeOneOrXnMinusOne<R, 32>` — uses both `(X - 2)`
  (for the linear C1/C2/C3/C6/C7 constraints) and `(X^32 - 1)` (for
  the cyclic-shift C4/C5 constraints). This is the only UAIR in the
  codebase that mixes ideal types within a single constraint system.
- **Constraints (user `t`-frame):**
  ```text
  C1: bp[0][t+1] + bp[1][t-3] - bp[2][t]   - bp[3][t]  - bp[4][t]
                - bp[5][t]    - bp[6][t]   - int[0][t] - int[1][t]   ∈ (X - 2)
  C2: bp[1][t+1] - bp[0][t-3] - bp[1][t-3] - bp[2][t]  - bp[3][t]
                - bp[7][t]    - int[0][t]  - int[2][t]                ∈ (X - 2)
  C3: bp[4][t]   - bp[4][t-16]- bp[4][t-7] - bp[8][t]  - bp[9][t]
                - int[3][t]                                            ∈ (X - 2)
  C4: bp[0][t] * (X^25 + X^14) - bp[5][t]                              ∈ (X^32 - 1)
  C5: bp[1][t] * (X^25 + X^13) - bp[3][t]                              ∈ (X^32 - 1)
  C6: bp[10][t] - int[0][t]                                            ∈ (X - 2)
  C7: bp[11][t] - int[1][t]                                            ∈ (X - 2)
  ```
  Note the **`+` sign on `bp[1][t-3]` in C1** (everything else in C1's
  RHS is subtracted). This single sign flip is necessary to break a
  boundary cascade in the C1 slack construction — see the trace section
  below for details.
- **Constraints (protocol `s`-frame, after `s = t - 16`):** every
  reference is now a forward shift. The only unshifted reference is
  `bp[4][s]` in C3 (it comes from `up.binary_poly[4]`); everything else
  goes through `down`.
  ```text
  C1: bp[0][s+17] + bp[1][s+13] - bp[2][s+16] - bp[3][s+16] - bp[4][s+16]
                 - bp[5][s+16]  - bp[6][s+16] - int[0][s+16] - int[1][s+16] ∈ (X-2)
  C2: bp[1][s+17] - bp[0][s+13] - bp[1][s+13] - bp[2][s+16] - bp[3][s+16]
                 - bp[7][s+16]  - int[0][s+16] - int[2][s+16]                ∈ (X-2)
  C3: bp[4][s+16] - bp[4][s]    - bp[4][s+9]  - bp[8][s+16] - bp[9][s+16]
                 - int[3][s+16]                                              ∈ (X-2)
  C4: bp[0][s+16] * (X^25 + X^14) - bp[5][s+16]                              ∈ (X^32 - 1)
  C5: bp[1][s+16] * (X^25 + X^13) - bp[3][s+16]                              ∈ (X^32 - 1)
  C6: bp[10][s+16] - int[0][s+16]                                            ∈ (X-2)
  C7: bp[11][s+16] - int[1][s+16]                                            ∈ (X-2)
  ```
  C4 and C5 are encoded via `mbs(&down.binary_poly[k], &x25_plus_x14)`
  (resp. `&x25_plus_x13`) where the scalars are `DensePolynomial<R, 32>`
  values with non-zero coefficients at the indicated positions. The
  result of `mbs` is a `DynamicPolynomialF<F>` of degree up to 56 —
  `mbs` does not truncate, since the underlying expression type grows
  dynamically. The membership check then folds the polynomial mod
  `X^32 - 1` (a "cyclic shift" interpretation: multiplying by `X^k` in
  `R[X]/(X^32 - 1)` is a cyclic left-shift by `k` of the coefficient
  vector).
- **Slacks for non-trivial witness construction.** Three columns appear
  in *exactly one* constraint and so can serve as free slacks that
  absorb the row's residue:
  - **`int[2]` slacks C2** (signed `i64`).
  - **`int[3]` slacks C3** (signed `i64`).
  - **`bp[6]` slacks C1** (a `BinaryPoly<32>`, so its `eval(2)` must be
    a non-negative `u32`).
- **Trace: row-by-row construction using the slacks.**
  1. **Pin `bp[0]` and `bp[1]` to large constants.** `bp[0][r] = 2^31`
     (single bit at position 31) and `bp[1][r] = 2^30` (single bit at
     position 30) at every row, except for the boundary tail (see
     below). Single-bit patterns trivially satisfy C4/C5's bit-gap
     constraint (no two 1-bits 11 or 12 apart) since there's only one
     bit.
  2. **Derive `bp[5]` and `bp[3]` from C4/C5** as the cyclic shifts:
     - `bp[5][r] = 2^24 + 2^13` (bp[0] cyclically shifted by 25 and 14:
       `(31+25) mod 32 = 24`, `(31+14) mod 32 = 13`).
     - `bp[3][r] = 2^23 + 2^11` (similar for bp[1]).
  3. **Random fillers.** `bp[2], bp[4], bp[7], bp[8], bp[9], bp[10],
     bp[11]` are 24-bit random binary polynomials. `bp[12]`, `bp[13]`
     are unconstrained random fillers.
  4. **Pin int[0]/int[1] from C6/C7.** `int[0][r] = bp[10][r].eval(2)`,
     `int[1][r] = bp[11][r].eval(2)`.
  5. **Compute slacks.** For each row, set `int[2][r]`, `int[3][r]`,
     and `bp[6][r]` to whatever value makes C2/C3/C1 hold at that row.
     `bp[6]` is constrained to `[0, 2^32 - 1]`; the other two are
     signed and can be any `i64`. The dominant positive terms in the
     C1 slack equation (`bp[0][r+1] = 2^31` and `bp[1][r-3] = 2^30`)
     keep `bp[6]`'s required value comfortably non-negative for normal
     rows.
- **Boundary handling.** The protocol enforces every constraint at
  rows `s ∈ [0, len-2]`. Slack columns are themselves shifted by 16,
  so `int[2]`, `int[3]`, and `bp[6]` are only in-bounds (and so usable
  as slacks) for `s ≤ len-17`. For `s ∈ [len-16, len-2]` the slack
  columns read as zero-padded, and the constraint must vanish without
  slack absorption. The witness handles this by zeroing certain
  columns in the boundary tail:
  - **C2 boundary** (`s ∈ [len-16, len-14]`): `bp[0][s+13]` and
    `bp[1][s+13]` are still in-bounds, but everything else is OOB. To
    make C2 vanish trivially, the witness zeros `bp[0]` and `bp[1]` on
    their last 3 rows (`r ∈ [len-3, len-1]`). C4/C5 then force
    `bp[5]`/`bp[3]` to zero on the same rows.
  - **C3 boundary** (`s ∈ [len-16, len-2]`): C3 reads `bp[4][s+16]`
    (OOB), `bp[4][s]` (in-bounds), `bp[4][s+9]` (in-bounds for
    `s ≤ len-10`). For the constraint to vanish, both in-bounds
    references must be zero. The witness zeros `bp[4]` on its last
    16 rows (`r ∈ [len-16, len-1]`).
  - **C1 boundary**: C1 at `s = len-17` has `bp[0][s+17]` OOB but
    `bp[1][s+13]` in-bounds. Without the `+` sign on `bp[1][t-3]`, the
    `bp[6]` slack would have to absorb a *negative* value here, which
    a `BinaryPoly<32>` cannot represent. The sign flip makes
    `bp[6][len-1]` the *positive* value `bp[1][len-4] − small ≈ 2^30`,
    which fits. For `s ∈ [len-16, len-14]` (where `bp[6]` is OOB), C1
    relies on the bp[0]/bp[1] zeroing from the C2 fix.

## Quick reference

Column counts are listed as `BinPoly / ArbPoly / Int`, matching the
`(binary_poly, arbitrary_poly, int)` ordering used by
`TotalColumnLayout::new`. "Shifts" are written as `count × amount`.

| UAIR | BinPoly | ArbPoly | Int | Shifts | Max degree | Ideal | Role |
|---|---:|---:|---:|---|---:|---|---|
| `BigLinearUair`                 | 16 | 0 | 1 | 16×1      | 1 | `(X-1), (X-2)`    | Wide linear benchmark (popcount-preserving) |
| `BigLinearUairWithPublicInput`  | 16 | 0 | 1 | 16×1      | 1 | `(X-1), (X-2)`    | Same as above + 4 public bp columns |
| `SHAProxy`                      | 14 | 0 | 4 | 21 specs (9, 13, 16, 17) | 1 | `(X-2), (X^32-1)` | Wide-shift mixed-ideal benchmark with multi-term `mbs` scalars |
