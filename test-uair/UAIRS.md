# UAIRs in `zinc-test-uair`

This crate defines the UAIRs (Univariate AIR-style constraint systems) that
the prover, verifier, PIOP, and end-to-end benchmarks exercise. Each UAIR is
a self-contained example: a column layout, optional row shifts, a constraint
function, and a random-trace generator. The constraints themselves are
written against the `ConstraintBuilder` DSL тАФ the prover and verifier each
provide their own builder implementations, so the same constraint code is
reused for both witness generation and verification.

This document is a reference for every UAIR currently in
[src/lib.rs](src/lib.rs): what its shape looks like, what it constrains,
what its trace generator produces, and what part of the system it exists to
exercise.

## Conventions

### Column kinds

A `TotalColumnLayout::new(b, a, i)` declares `b + a + i` columns of three
distinct kinds. Each column kind stores a different value type per row:

- **Binary-polynomial columns** (`bp`) hold `BinaryPoly<32>` тАФ a polynomial
  with 32 coefficients, each restricted to `0` or `1`. Note that the
  *coefficients* are binary; the column entry itself is still a polynomial,
  not a single bit. Useful identities: `bp.eval(1)` is the popcount (number
  of `1` coefficients) and `bp.eval(2)` is the integer encoded by reading
  the coefficients as the bits of a `u32`.
- **Arbitrary-polynomial columns** (`ap`) hold `DensePolynomial<R, 32>` тАФ a
  polynomial of degree `< 32` with arbitrary coefficients in the host ring
  `R`.
- **Integer columns** (`int`) hold a single `R` element per row (a scalar,
  with no polynomial structure).

Throughout this document, `bp[i]`, `ap[i]`, and `int[i]` refer to the
`i`-th column of each kind.

### Rows: `up` and `down`

The constraint function receives two row views per call:

- `up.X[i]` is the value of column `X[i]` at the current row `t`.
- `down.X[i]` is the value of column `X[i]` at the *shifted* row.

**Important тАФ `down` is indexed by shift-spec position, not by source
column.** The signature declares a `Vec<ShiftSpec>`; the `j`-th entry of
`down.binary_poly` corresponds to the `j`-th `ShiftSpec` in that vector,
not to `bp[j]`. Concrete examples:

- `BigLinearUair` uses `(0..16).map(|i| ShiftSpec::new(i, 1))`. The shift
  vector is in source-column order, so `down.binary_poly[i]` happens to
  equal `bp[i][t+1]` here.
- `SHAProxy` uses 21 shift specs (17 binary-poly + 4 int) with shift
  amounts of 9, 13, 16, and 17 тАФ including **multiple shift amounts on
  the same source column** (`bp[0]` and `bp[1]` each get three shifts;
  `bp[4]` gets two). This is the only UAIR in the codebase that mixes
  binary-poly and int shifts in the same shift vec, and the only one
  where the same `source_col` appears with multiple amounts.
  `UairSignature::new` automatically sorts the shift vec stably by
  `source_col`, so writing the shift specs in source-column order in the
  signature makes the `down.binary_poly[k]` indexing in the constraint
  code mechanical to derive.

Shifts are zero-padded beyond the trace boundary: the "virtual" row past
the last real row reads as all zeros.

### Constraints

A `ConstraintBuilder` exposes two assertion primitives:

- `b.assert_zero(expr)` тАФ the expression must reduce to the zero element.
- `b.assert_in_ideal(expr, &ideal)` тАФ the expression must lie in the given
  ideal of `R[X]`. At verification time, expressions are projected to a
  prime field and the membership check is evaluated there.

Expressions are built from column values using `+`, `-`, `*`, plus two
extras supplied to the constraint function:

- `from_ref(&scalar)` lifts a fixed `Self::Scalar` (a
  `DensePolynomial<R, 32>`) into the expression type тАФ i.e., a constant
  polynomial term.
- `mbs(&expr, &scalar)` multiplies an existing expression by a fixed
  `Self::Scalar`. This is the only way to multiply a column expression by
  a polynomial constant such as `X` (encoded as `DensePolynomial::new([0, 1])`).

### Ideal types

Two ideal implementations ship today:

- **`DegreeOneIdeal<R>`** generated by `X - r` for some root `r тИИ R`. An
  expression `e(X)` lies in `(X - r)` iff `e(r) = 0` (i.e., the projected
  polynomial vanishes at `r` in the field). This document writes such
  ideals as `(X - r)`. The two roots used in practice are `r = 1` (so
  `(X - 1)` checks "popcount/sum" relations) and `r = 2` (so `(X - 2)`
  checks "integer-decomposition" relations).
- **`XnMinusOneIdeal<N>`** generated by `X^N - 1`. An expression
  `e(X) тИИ R[X]` lies in `(X^N - 1)` iff its coefficients fold to zero
  modulo `N`: for each `k тИИ 0..N`, `╬г_{i тЙб k (mod N)} e_i = 0` in the
  field. Geometrically, `R[X]/(X^N - 1)` is the ring where multiplying
  by `X^k` is a cyclic left-shift of the coefficient vector by `k`,
  which makes this ideal a natural fit for cyclic-bit-shuffle relations
  (e.g. SHA-style ╧Г/╬г functions). Currently used only by `SHAProxy`
  with `N = 32`.
- **`MixedDegreeOneOrXnMinusOne<R, N>`** is an enum that wraps either
  a `DegreeOneIdeal<R>` or an `XnMinusOneIdeal<N>`. UAIRs that need
  both kinds of ideals in the same constraint system use this as
  `Self::Ideal`. Currently used only by `SHAProxy` with `N = 32`.
- **`ImpossibleIdeal`** has `contains(_) = false`. It is the placeholder
  type used by UAIRs whose constraints are exclusively `assert_zero`
  (no ideal-membership check is ever performed against this type).

## Why every ECDSA UAIR in this crate is non-satisfying

`EcdsaUair`, `EcdsaUairLimbs`, and `ShaProxyEcdsaUair` all encode their
constraints as **exact integer equalities** (every constraint is
`assert_zero` over the cell type) and generate witnesses by performing a
real Shamir's-trick walk over secp256k1 with **modular** F_p arithmetic.
The two are inconsistent on purpose, and the resulting witnesses fail to
satisfy the constraint system. The benches are therefore **prover-only**
тАФ running the verifier on any of these traces would correctly reject it.

This document records why we never closed that gap, because it looks
like an obvious bug at first glance and will keep looking like one to
future readers. Two distinct issues block a "real walk + satisfying
witness" combination, and **both** would have to be solved together.

### Issue 1: integer arithmetic is not modular arithmetic

The constraint `C1: Z_mid тИТ 2┬╖Y┬╖Z = 0` says, as written, that the cell
`Z_mid` equals the integer `2┬╖Y┬╖Z` exactly. The witness builds `Z_mid`
via `fp_mul`, so `Z_mid = (2┬╖Y┬╖Z) mod P` where `P` is the secp256k1
base prime (`тЙИ 2┬▓тБ╡тБ╢`). For Y, Z near P, the integer product `2┬╖Y┬╖Z` is
close to `2┬╖P┬▓` and the modular reduction is non-trivial: `Z_mid` and
`2┬╖Y┬╖Z` differ by some multiple of P, not by zero. The constraint as
written checks integer equality, the witness checks modular equality,
and these are different relations.

The standard fix for this kind of arithmetization gap is to add a
**quotient column** `q` per constraint, rewriting `lhs = 0 (mod P)` as
`lhs тИТ q┬╖P = 0 (as exact integers)`. The witness sets `q = lhs/P` so
the equation holds exactly, and the prover (which evaluates over a
random PIOP field `F`) sees the integer identity preserved because
projection from тДд to `F` is a ring homomorphism. This is well-known and
straightforward in principle. In practice, see Issue 2.

### Issue 2: `Int<4>` projects to `MontyField` with sign-aware semantics

The trace cells are `Int<4>` (256-bit signed integers), and the prover
projects them to its PIOP field `F` (a `MontyField`) via
`FromWithConfig<&Int<N>>`. That impl, in
[`crypto-primitives`](https://github.com/NethermindEth/crypto-primitives.git),
literally does:

```rust
let mut abs = value.inner().abs();
// ... compute MontyField representation of |value| ...
if value.is_negative() { -result } else { result }
```

It takes the **absolute value** of the underlying `Int<4>` and negates
if `is_negative()` (which checks the bit-255 sign bit). This is
mathematically correct for integers but fights us when we want to use
`Int<4>` as a *256-bit container* for canonical F_p representatives in
`[0, P)`.

Concretely: roughly **half** of all canonical F_p elements have bit
255 set (since `P тЙИ 2┬▓тБ╡тБ╢ тИТ 2┬│┬▓ тИТ 977 > 2┬▓тБ╡тБ╡`). For each such element,
the bit pattern stored in the `Int<4>` cell looks "negative" to
`MontyField`. For instance, the canonical value `P тИТ 1 тЙИ 2┬▓тБ╡тБ╢ тИТ 2┬│┬▓ тИТ
978` projects to `MontyField(тИТ(2┬│┬▓ + 978))`, **not** to `MontyField(P тИТ
1)`. These are different field elements (they agree mod `P`, but not
mod the PIOP field's modulus, which is what `MontyField` actually
checks).

Consequence: even if we add quotient columns to fix Issue 1, the
resulting "satisfying" integer identity does not project to a
satisfying `F`-identity, because the sign of about half of the cell
projections is wrong relative to what the witness intended. The
quotient trick produces an integer identity that's only satisfied
under one interpretation of the bit pattern (unsigned тДд embedded in
тДд), and `MontyField` uses a different one (signed `Int<4>` interpreted
in two's complement and then sign-extended into `F`).

### So why did we ever try?

The first version of `EcdsaUair` was designed before we understood this
projection behavior. At the time, the question seemed to be a simple
choice between "real walk, accept non-satisfying" and "all-zero fixed
point, satisfying but trivial" тАФ and the user picked the real walk for
trace-shape realism. The plan was always that we could later add
quotient columns to make the real walk satisfying.

That later attempt is what produced this document. Discovering Issue 2
mid-design made it clear that closing the gap requires more than
quotient columns:

- **Switch to limb form** (`i64` cells holding 32-bit u32 limbs, like
  [`EcdsaUairLimbs`](#ecdsauairlimbs)) so that every cell is small and
  positive and the sign-aware projection is unambiguous. Then add
  **quotient + carry columns** to do per-limb modular reduction. This
  is essentially the production approach used by real ECDSA UAIRs in
  other zk libraries; estimated cost is hundreds of new columns and
  ~1000+ lines of constraint code.
- **Switch to `Int<8>` cells** so the canonical 256-bit values fit as
  positive 512-bit signed integers (sign bit is bit 511, never set for
  values in `[0, 2┬▓тБ╡тБ╢)`). This solves Issue 2 cleanly with one cell per
  field element, but requires a new `ZincTypes` config and bench
  helper, and pays a per-cell PCS cost penalty.
- **Toy curve over a small prime** (e.g. FтВБтВАтВБ): all field elements
  trivially fit in `i64`, both issues vanish, and the constraint shape
  is preserved. This is the approach taken by v1's `EcdsaUairDp`. The
  downside is that it stops being secp256k1.

Each of these is its own substantial project; none is a small patch on
the existing UAIRs. We chose to leave the current ECDSA UAIRs as
**prover-only benchmark fixtures** with an explicit "non-satisfying
witness" caveat in their per-UAIR sections, rather than fix the gap
in-place.

The prover-only benches (`bench_ecdsa`, `bench_ecdsa_limbs`,
`bench_sha_ecdsa`) still measure realistic prover throughput on
realistic constraint shapes тАФ they just don't exercise the verifier.
For verifier-running benches we use the satisfying UAIRs in this crate
(`BigLinearUair`, `BigLinearUairWithPublicInput`, `SHAProxy`, etc.).

## Benchmark UAIRs

These UAIRs are sized for the end-to-end criterion benchmark in
[protocol/benches/e2e.rs](../protocol/benches/e2e.rs). They are deliberately
"wide" (many columns and many constraints) so that prover/verifier
performance numbers reflect realistic per-constraint and per-column
overheads rather than fixed-cost setup.

### `BigLinearUair<R>`

A many-column linear UAIR encoding a *popcount-preservation* relation:
each binary-polynomial column is allowed to permute its `1` bits across
rows, but its popcount must stay constant; in addition, column 0 is
forced to be the `BinaryPoly` representation of the row's total popcount
across all 16 columns.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Shifts:** all 16 bp columns shifted by 1 (so `down.binary_poly[i] =
  bp[i][t+1]`, taking advantage of the source-column order in the shift
  vector)
- **Ideal:** `(X - 1)` and `(X - 2)`
- **Constraints (per row `t`):**
  ```
  (bp[0] + bp[1] + тАж + bp[15]) - int[0]            тИИ (X - 1)   // int[0] = total popcount
  down.binary_poly[0]          - int[0]            тИИ (X - 2)   // bp[0][t+1] encodes that popcount as a u32
  for i in 1..16:
      up.binary_poly[i]        - down.binary_poly[i] тИИ (X - 1) // bp[i] preserves popcount across rows
  ```
- **Trace:** row 0 is fully random; for each subsequent row, `bp[1..16]`
  are fresh random binary polynomials that *match the previous row's
  popcount on each column*, and `bp[0][t+1]` is set to
  `BinaryPoly::from(total_popcount)` so that the second constraint holds.
- **Why it exists:** stresses the prover's per-column and per-constraint
  cost (16 columns ├Ч 17 constraints, all degree 1) without introducing any
  non-linearity. The matching MLE-first prover path also runs because the
  max constraint degree is 1.

### `BigLinearUairWithPublicInput<R>`

Identical to `BigLinearUair` (same layout, shifts, constraints, trace),
but the first 4 binary-polynomial columns are marked as public input.
Exists so the end-to-end benchmark can also measure the public-input
plumbing on the same constraint shape.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Public layout:** `(4 bp, 0 ap, 0 int)` тАФ first 4 bp columns are public
- **Shifts / ideal / constraints / trace:** delegated to `BigLinearUair`

### `SHAProxy<R>`

A "wide and shifty" linear UAIR designed to stress the protocol's
shift-handling, `mbs` machinery, and **mixed-ideal** support. It has
14 binary-polynomial columns, 4 integer columns, **21 shift specs**
(17 bp + 4 int) with **multiple shift amounts on the same source
column**, **scalar polynomial multiplications** with multi-term,
high-degree scalars (`X^25 + X^14` and `X^25 + X^13`), and uses both
`(X - 2)` and `(X^32 - 1)` ideal types in the same UAIR.

The name reflects the inspiration тАФ the constraint shape is reminiscent
of bit/integer relations found in proxies for SHA-style hash circuits
(specifically, the cyclic-shift-and-mix structure of SHA's ╧Г/╬г
functions) тАФ even though SHAProxy does not actually compute a SHA
round.

The user-facing constraints are written in a `t`-relative frame with
**negative shifts** (`bp[k][t-N]`). Since `ShiftSpec` only supports
forward shifts, the constraint set is **globally re-indexed** by
substituting `s = t - 16` so that every reference becomes a forward
shift in `s`. The constraint *shape* (which columns interact, with what
coefficients) is preserved exactly. Maximum forward shift in the
`s`-frame is 17.

- **Layout:** `(14 bp, 0 ap, 4 int)`
- **Shifts:** 21 specs total (17 binary-poly + 4 int), with shift
  amounts of 9, 13, 16, and 17. Multiple shifts per source column:
  - `bp[0]`: shifts 13, 16, 17 тЖТ `down.binary_poly[0..3]`
  - `bp[1]`: shifts 13, 16, 17 тЖТ `down.binary_poly[3..6]`
  - `bp[2]`, `bp[3]`: shift 16 тЖТ `down.binary_poly[6..8]`
  - `bp[4]`: shifts 9, 16 тЖТ `down.binary_poly[8..10]`
  - `bp[5..=11]`: shift 16 тЖТ `down.binary_poly[10..17]`
  - `int[0..=3]`: shift 16 тЖТ `down.int[0..4]`
  - `bp[12]` and `bp[13]` are unused fillers (no shift, no constraint).
- **Ideal:** `MixedDegreeOneOrXnMinusOne<R, 32>` тАФ uses both `(X - 2)`
  (for the linear C1/C2/C3/C6/C7 constraints) and `(X^32 - 1)` (for
  the cyclic-shift C4/C5 constraints). This is the only UAIR in the
  codebase that mixes ideal types within a single constraint system.
- **Constraints (user `t`-frame):**
  ```text
  C1: bp[0][t+1] + bp[1][t-3] - bp[2][t]   - bp[3][t]  - bp[4][t]
                - bp[5][t]    - bp[6][t]   - int[0][t] - int[1][t]   тИИ (X - 2)
  C2: bp[1][t+1] - bp[0][t-3] - bp[1][t-3] - bp[2][t]  - bp[3][t]
                - bp[7][t]    - int[0][t]  - int[2][t]                тИИ (X - 2)
  C3: bp[4][t]   - bp[4][t-16]- bp[4][t-7] - bp[8][t]  - bp[9][t]
                - int[3][t]                                            тИИ (X - 2)
  C4: bp[0][t] * (X^25 + X^14) - bp[5][t]                              тИИ (X^32 - 1)
  C5: bp[1][t] * (X^25 + X^13) - bp[3][t]                              тИИ (X^32 - 1)
  C6: bp[10][t] - int[0][t]                                            тИИ (X - 2)
  C7: bp[11][t] - int[1][t]                                            тИИ (X - 2)
  ```
  Note the **`+` sign on `bp[1][t-3]` in C1** (everything else in C1's
  RHS is subtracted). This single sign flip is necessary to break a
  boundary cascade in the C1 slack construction тАФ see the trace section
  below for details.
- **Constraints (protocol `s`-frame, after `s = t - 16`):** every
  reference is now a forward shift. The only unshifted reference is
  `bp[4][s]` in C3 (it comes from `up.binary_poly[4]`); everything else
  goes through `down`.
  ```text
  C1: bp[0][s+17] + bp[1][s+13] - bp[2][s+16] - bp[3][s+16] - bp[4][s+16]
                 - bp[5][s+16]  - bp[6][s+16] - int[0][s+16] - int[1][s+16] тИИ (X-2)
  C2: bp[1][s+17] - bp[0][s+13] - bp[1][s+13] - bp[2][s+16] - bp[3][s+16]
                 - bp[7][s+16]  - int[0][s+16] - int[2][s+16]                тИИ (X-2)
  C3: bp[4][s+16] - bp[4][s]    - bp[4][s+9]  - bp[8][s+16] - bp[9][s+16]
                 - int[3][s+16]                                              тИИ (X-2)
  C4: bp[0][s+16] * (X^25 + X^14) - bp[5][s+16]                              тИИ (X^32 - 1)
  C5: bp[1][s+16] * (X^25 + X^13) - bp[3][s+16]                              тИИ (X^32 - 1)
  C6: bp[10][s+16] - int[0][s+16]                                            тИИ (X-2)
  C7: bp[11][s+16] - int[1][s+16]                                            тИИ (X-2)
  ```
  C4 and C5 are encoded via `mbs(&down.binary_poly[k], &x25_plus_x14)`
  (resp. `&x25_plus_x13`) where the scalars are `DensePolynomial<R, 32>`
  values with non-zero coefficients at the indicated positions. The
  result of `mbs` is a `DynamicPolynomialF<F>` of degree up to 56 тАФ
  `mbs` does not truncate, since the underlying expression type grows
  dynamically. The membership check then folds the polynomial mod
  `X^32 - 1` (a "cyclic shift" interpretation: multiplying by `X^k` in
  `R[X]/(X^32 - 1)` is a cyclic left-shift by `k` of the coefficient
  vector).
- **Slacks for non-trivial witness construction.** Three columns appear
  in *exactly one* constraint and so can serve as free slacks that
  absorb the row's residue:
  - **`int[2]` slacks C2** (signed `i64`).
  - **`int[3]` slacks C3** (signed `i64`).
  - **`bp[6]` slacks C1** (a `BinaryPoly<32>`, so its `eval(2)` must be
    a non-negative `u32`).
- **Trace: row-by-row construction using the slacks.**
  1. **Pin `bp[0]` and `bp[1]` to large constants.** `bp[0][r] = 2^31`
     (single bit at position 31) and `bp[1][r] = 2^30` (single bit at
     position 30) at every row, except for the boundary tail (see
     below). Single-bit patterns trivially satisfy C4/C5's bit-gap
     constraint (no two 1-bits 11 or 12 apart) since there's only one
     bit.
  2. **Derive `bp[5]` and `bp[3]` from C4/C5** as the cyclic shifts:
     - `bp[5][r] = 2^24 + 2^13` (bp[0] cyclically shifted by 25 and 14:
       `(31+25) mod 32 = 24`, `(31+14) mod 32 = 13`).
     - `bp[3][r] = 2^23 + 2^11` (similar for bp[1]).
  3. **Random fillers.** `bp[2], bp[4], bp[7], bp[8], bp[9], bp[10],
     bp[11]` are 24-bit random binary polynomials. `bp[12]`, `bp[13]`
     are unconstrained random fillers.
  4. **Pin int[0]/int[1] from C6/C7.** `int[0][r] = bp[10][r].eval(2)`,
     `int[1][r] = bp[11][r].eval(2)`.
  5. **Compute slacks.** For each row, set `int[2][r]`, `int[3][r]`,
     and `bp[6][r]` to whatever value makes C2/C3/C1 hold at that row.
     `bp[6]` is constrained to `[0, 2^32 - 1]`; the other two are
     signed and can be any `i64`. The dominant positive terms in the
     C1 slack equation (`bp[0][r+1] = 2^31` and `bp[1][r-3] = 2^30`)
     keep `bp[6]`'s required value comfortably non-negative for normal
     rows.
- **Boundary handling.** The protocol enforces every constraint at
  rows `s тИИ [0, len-2]`. Slack columns are themselves shifted by 16,
  so `int[2]`, `int[3]`, and `bp[6]` are only in-bounds (and so usable
  as slacks) for `s тЙд len-17`. For `s тИИ [len-16, len-2]` the slack
  columns read as zero-padded, and the constraint must vanish without
  slack absorption. The witness handles this by zeroing certain
  columns in the boundary tail:
  - **C2 boundary** (`s тИИ [len-16, len-14]`): `bp[0][s+13]` and
    `bp[1][s+13]` are still in-bounds, but everything else is OOB. To
    make C2 vanish trivially, the witness zeros `bp[0]` and `bp[1]` on
    their last 3 rows (`r тИИ [len-3, len-1]`). C4/C5 then force
    `bp[5]`/`bp[3]` to zero on the same rows.
  - **C3 boundary** (`s тИИ [len-16, len-2]`): C3 reads `bp[4][s+16]`
    (OOB), `bp[4][s]` (in-bounds), `bp[4][s+9]` (in-bounds for
    `s тЙд len-10`). For the constraint to vanish, both in-bounds
    references must be zero. The witness zeros `bp[4]` on its last
    16 rows (`r тИИ [len-16, len-1]`).
  - **C1 boundary**: C1 at `s = len-17` has `bp[0][s+17]` OOB but
    `bp[1][s+13]` in-bounds. Without the `+` sign on `bp[1][t-3]`, the
    `bp[6]` slack would have to absorb a *negative* value here, which
    a `BinaryPoly<32>` cannot represent. The sign flip makes
    `bp[6][len-1]` the *positive* value `bp[1][len-4] тИТ small тЙИ 2^30`,
    which fits. For `s тИИ [len-16, len-14]` (where `bp[6]` is OOB), C1
    relies on the bp[0]/bp[1] zeroing from the C2 fix.

### `EcdsaUair`

A 14-column / 258-row arithmetization of one secp256k1 ECDSA signature
verification using **Shamir's trick** for the simultaneous scalar
multiplication `uтВБ┬╖G + uтВВ┬╖Q`. The constraint shape is inspired by ┬з6 of
the agentic-approach-v1 `IMPLEMENTATION.md`: it stresses the prover with a
**non-linear, multi-shift** constraint set over **256-bit signed integers**
тАФ the only UAIR in this crate that uses `Int<4>` as the trace cell type
instead of `BinaryPoly<32>` or a small ring.

The doubling/addition formulas for short Weierstrass `a = 0` (secp256k1)
naturally produce a constraint with a degree-10 sub-expression (`R_a┬▓`,
where `R_a = T_y┬╖Z_mid┬│ тИТ Y_mid` is the chord-slope numerator). To keep
the maximum constraint degree low, this UAIR commits **three wire
columns** (`R_a`, `H┬▓`, `H┬│`) so the addition-step constraints can read
those quantities at degree 1 instead of recomputing them inline. The
final maximum degree is **5**.

Unlike every other UAIR in this crate, **this one is not generic over `R`**.
It is hard-coded to `Int<4>` (256-bit signed) because the secp256k1 base
field constants (`p`, `Gx`, `Gy`, `Q = 2G`, `G+Q = 3G`) only fit in 256
bits. The trace cells are stored as `Int<4>` via the bit-pattern reinterpret
of `Uint<4>` field elements (values in `[2┬│┬╣тБ╡, 2┬│тБ╡тБ╢)` appear as negative
`Int<4>` values, which is harmless because the witness is non-satisfying тАФ
see below).

- **Layout:** `(0 bp, 0 ap, 14 int)`
- **Public layout:** `(0 bp, 0 ap, 4 int)` тАФ `bтВБ`, `bтВВ`, `sel_init`,
  `sel_final` are public; the verifier scans them directly. The
  public-columns-precede-witness convention forces this exact int ordering:

  | int idx | Name        | Public | Description                                       |
  |--------:|-------------|:------:|---------------------------------------------------|
  |       0 | `bтВБ`        |   тЬУ    | Bit of `uтВБ = e┬╖sтБ╗┬╣ mod n` (selected for the row)  |
  |       1 | `bтВВ`        |   тЬУ    | Bit of `uтВВ = r┬╖sтБ╗┬╣ mod n`                         |
  |       2 | `sel_init`  |   тЬУ    | `1` at row 0, `0` elsewhere                       |
  |       3 | `sel_final` |   тЬУ    | `1` at row 257, `0` elsewhere                     |
  |       4 | `X`         |        | Accumulator x (Jacobian) тАФ shift source           |
  |       5 | `Y`         |        | Accumulator y (Jacobian) тАФ shift source           |
  |       6 | `Z`         |        | Accumulator z (Jacobian) тАФ shift source           |
  |       7 | `X_mid`     |        | Result of doubling step                           |
  |       8 | `Y_mid`     |        |                                                   |
  |       9 | `Z_mid`     |        |                                                   |
  |      10 | `H`         |        | Addition scratch (`T_x ┬╖ Z_mid┬▓ тИТ X_mid`)         |
  |      11 | `R_a`       |        | **Wire**: `T_y ┬╖ Z_mid┬│ тИТ Y_mid` (chord numerator) |
  |      12 | `H_sq`      |        | **Wire**: `H┬▓`                                    |
  |      13 | `H_cu`      |        | **Wire**: `H┬│`                                    |

- **Shifts:** 3 forward shifts of amount 1 on the accumulator coordinates
  `X`, `Y`, `Z` (flat indices 4, 5, 6). After `UairSignature::new`'s
  stable sort by `source_col`, `down.int[0..3]` map to `X[t+1]`, `Y[t+1]`,
  `Z[t+1]` respectively (in source-column order, like
  `BigLinearUair`).
- **Ideal:** `ImpossibleIdeal` тАФ every constraint is `assert_zero`, no
  ideal-membership check is ever performed. This is the placeholder pattern
  used by `TestUairSimpleMultiplication`.
- **Selectors and gating.** All 12 constraints are unconditional from the
  framework's perspective; per-row activation for the *boundary* checks is
  encoded *inside the expressions* via the public selector columns:
  - `sel_init` is `1` at row 0 and `0` elsewhere тАФ used to make B3 fire
    only at the first row.
  - `sel_final` is `1` at row 257 (the last "real" row) and `0`
    elsewhere тАФ used to make B4 fire only at the boundary.
  The step constraints C5тАУC7 are **not** gated by `sel_final` (an earlier
  draft did this to drop one degree, but the resulting witness is
  non-satisfying anyway, so the gate was removed in favour of cleaner
  degree-5 expressions).
- **Inlined sub-expressions still used inside the constraints.** Two
  quantities are written directly into the constraint expressions instead
  of being committed as columns, because their natural degrees (1 and 2)
  do not threaten the max-degree budget:
  - **Shamir indicator** `s = bтВБ + bтВВ тИТ bтВБ┬╖bтВВ` тАФ equals `1` iff at least
    one of `bтВБ, bтВВ` is set. Degree 2 in the public bit columns.
  - **Table point** `(T_x, T_y)` selected from `{O, G, Q, G+Q}` by
    `(bтВБ, bтВВ)`:
    ```
    T_x = (1тИТbтВБ) bтВВ ┬╖ Qx + bтВБ (1тИТbтВВ) ┬╖ Gx + bтВБ bтВВ ┬╖ GQx
    T_y = (1тИТbтВБ) bтВВ ┬╖ Qy + bтВБ (1тИТbтВВ) ┬╖ Gy + bтВБ bтВВ ┬╖ GQy
    ```
    (the `(1тИТbтВБ)(1тИТbтВВ) тЖТ identity` case contributes 0 to both). Degree 2
    in `(bтВБ, bтВВ)`.

  Three other quantities тАФ `R_a`, `H┬▓`, `H┬│` тАФ used to be inlined too
  (and pushed the max constraint degree to 13 because of `R_a┬▓`). They
  are now committed as wire columns instead, and W1/W2/W3 below tie them
  to their definitions.

  The secp256k1 base-field constants `Gx, Gy, Qx, Qy, GQx, GQy, R_SIG` are
  injected via `from_ref` as degree-0 `DensePolynomial<Int<4>, 32>`
  scalars. Note that this UAIR's `Self::Scalar = DensePolynomial<Int<4>, 32>`
  is an artifact of the `Uair` trait тАФ the polynomial structure is unused;
  every constant is a degree-0 polynomial (scalar `Int<4>`).

- **Constraints (12 total тАФ 4 doubling + 3 wire + 3 addition + 2 boundary):**
  ```
  // Doubling (a = 0 short Weierstrass formulas):
  C1: Z_mid тИТ 2┬╖Y┬╖Z                                              = 0
  C2: X_mid тИТ (9┬╖XтБ┤ тИТ 8┬╖X┬╖Y┬▓)                                    = 0
  C3: Y_mid тИТ (12┬╖X┬│┬╖Y┬▓ тИТ 3┬╖X┬▓┬╖X_mid тИТ 8┬╖YтБ┤)                     = 0

  // Addition scratch and the three wire columns:
  C4: H    тИТ (T_x ┬╖ Z_mid┬▓ тИТ X_mid)                              = 0
  W1: H_sq тИТ H ┬╖ H                                               = 0   // wires H┬▓
  W2: H_cu тИТ H_sq ┬╖ H                                            = 0   // wires H┬│ (chained, degree 2 not 3)
  W3: R_a  тИТ (T_y ┬╖ Z_mid┬│ тИТ Y_mid)                              = 0   // wires the chord numerator

  // Conditional add of T = table[(bтВБ, bтВВ)] using the wire columns:
  C5: Z[t+1] тИТ ((1тИТs)┬╖Z_mid + s ┬╖ Z_mid ┬╖ H)                     = 0
  C6: X[t+1] тИТ ((1тИТs)┬╖X_mid + s ┬╖ (R_a┬▓ тИТ H_cu тИТ 2┬╖X_mid┬╖H_sq))  = 0
  C7: Y[t+1] тИТ ((1тИТs)┬╖Y_mid + s ┬╖ (R_a┬╖(X_mid┬╖H_sq тИТ X[t+1])
                                    тИТ Y_mid┬╖H_cu))               = 0

  // Boundary checks:
  B3: sel_init ┬╖ Z                                               = 0   // accumulator = O at row 0
  B4: sel_final ┬╖ Z ┬╖ (X тИТ R_SIG ┬╖ Z┬▓)                           = 0   // affine x-coord = R_SIG at row 257
  ```
  Per-constraint degrees (as reported by
  `count_constraint_degrees::<EcdsaUair>()` and asserted in the test):
  `[2, 4, 5, 4, 2, 2, 5, 4, 4, 5, 2, 4]`. Maximum is **5**, set jointly
  by C3 (`12┬╖X┬│┬╖Y┬▓` from the doubling formula) and W3/C7 (the chord-slope
  path through `T_y ┬╖ Z_mid┬│`). C6's `R_a┬▓` is now degree 2 instead of
  10 because `R_a` is wired.

- **Trace generator.** A **real MSB-first double-and-add walk** over
  secp256k1 for random scalars `uтВБ, uтВВ`. For each row `r тИИ [0, 256)`:
  1. Read `(bтВБ[r], bтВВ[r])` = the `r`-th MSB of each scalar.
  2. Double the current accumulator over `F_p` to get `(X_mid, Y_mid, Z_mid)`.
  3. If at least one bit is set, point-add the table point
     `T = {O, G, Q, G+Q}[(bтВБ, bтВВ)]` to produce the next-row accumulator;
     otherwise pass `(X_mid, Y_mid, Z_mid)` through unchanged. `H` is
     stored in either case for self-consistency with C4.
  4. Compute the wire values from the field arithmetic: `R_a = T_y┬╖Z_mid┬│
     тИТ Y_mid (mod p)`, `H_sq = H┬▓ (mod p)`, `H_cu = H┬│ (mod p)`.
  5. Write all 12 data cells plus `bтВБ`, `bтВВ` into the trace.

  Row 256 holds the final accumulator (= `uтВБ┬╖G + uтВВ┬╖Q`) with junk
  doubling/H cells; row 257 is the boundary row holding the same
  accumulator and `sel_final = 1`. Rows 258 and beyond are padded with
  zeros to reach `1 << num_vars` total length. `num_vars тЙе 9` is
  required (since `258 тЙд 512 = 2тБ╣`).

- **The witness intentionally does NOT satisfy the constraints.** This is
  the critical thing to know about `EcdsaUair` and the reason it is the
  *only* UAIR in this crate where running the IC prover or the Zinc+
  verifier will reject the trace. The constraints (e.g. C1: `Z_mid тИТ
  2┬╖Y┬╖Z = 0`) are **exact integer equality** over `Int<4>`. The walk
  produces values that are reduced mod `p тЙИ 2┬▓тБ╡тБ╢`, so `2┬╖Y┬╖Z` and `Z_mid`
  agree only modulo `p`, not exactly тАФ and `2┬╖Y┬╖Z` overflows `Int<4>`
  before the implicit reduction even occurs. A satisfying witness for
  this exact constraint set would need either auxiliary quotient/carry
  columns (not present) or the trivial all-zero fixed-point trace (which
  would defeat the purpose of stressing the prover with realistic data).

  As a result, **this UAIR is intended for prover-only benchmarking**.
  See the `bench_ecdsa` function in [protocol/benches/e2e.rs](../protocol/benches/e2e.rs)
  for the prover-only criterion harness; do not run the verifier or the
  IC prover on this trace.

- **Why it exists:** stresses the prover on a realistic ECDSA-shaped
  constraint set тАФ 12 constraints, max degree 5, 256-bit `Int<4>` trace
  cells, 4 public columns, 3 forward shifts on the accumulator. It is
  the first UAIR in this crate to use `Int<4>` as the trace type, the
  first to combine non-linear constraints with selector-gated boundary
  conditions, and the only one that uses *wire columns* to flatten
  high-degree sub-expressions.

### `EcdsaUairLimbs`

A **limb-decomposed variant of `EcdsaUair`** where each 256-bit Jacobian
coordinate is split into **8 little-endian u32 limbs** stored in 8
separate witness columns. The constraint set mirrors `EcdsaUair`
one-for-one тАФ every `Int<4>` constraint is emitted as **8 per-limb
constraints** тАФ and uses the same wire columns (`R_a`, `H_sq`, `H_cu`)
to keep the maximum degree at 5. There are **no carry columns**: high
partial-product limbs are silently discarded, matching `Int<4>`'s
mod-`2┬▓тБ╡тБ╢` wrap-around. The witness inherits `EcdsaUair`'s
non-satisfaction property.

The cell type is `i64` (matching the existing `BenchZincTypes::Int =
i64`) so that a single `i32 * i32` partial product fits without wrapping
inside one cell. Limb *values* are u32-range (0..2┬│┬▓) but the *storage*
is 64-bit. The bench reuses the existing infrastructure with no new
ZincTypes scaffolding.

- **Layout:** `(0 bp, 0 ap, 84 int)` тАФ 4 public + 80 witness limb cells.
- **Public layout:** `(0, 0, 4)` тАФ `bтВБ`, `bтВВ`, `sel_init`, `sel_final`,
  same as `EcdsaUair`.
- **Int column ordering** (each labelled run is the 8 little-endian limbs
  of one 256-bit value):

  | Range | Logical value | Public? |
  |---:|---|:---:|
  | `0..4` | `bтВБ, bтВВ, sel_init, sel_final` | тЬУ |
  | `4..12` | `X` (shift source) | |
  | `12..20` | `Y` (shift source) | |
  | `20..28` | `Z` (shift source) | |
  | `28..36` | `X_mid` | |
  | `36..44` | `Y_mid` | |
  | `44..52` | `Z_mid` | |
  | `52..60` | `H` | |
  | `60..68` | `R_a` (wire) | |
  | `68..76` | `H_sq` (wire) | |
  | `76..84` | `H_cu` (wire) | |

- **Shifts:** **24** specs тАФ `X[k]`, `Y[k]`, `Z[k]` each shifted by 1 for
  `k = 0..8`. After `UairSignature::new`'s stable sort by `source_col`:
  `down.int[0..8] = X[t+1]`, `down.int[8..16] = Y[t+1]`, `down.int[16..24]
  = Z[t+1]`, all little-endian.
- **Ideal:** `DegreeOneIdeal<i64>` тАФ never used (every constraint is
  `assert_zero`). Picked for compatibility with `do_bench_uair`'s
  hard-coded ideal type, not because the UAIR ever calls `assert_in_ideal`.
- **Scalar:** `DensePolynomial<i64, 32>` тАФ matches `BenchZincTypes`.
- **Constraint code structure.** The body uses **inline closure helpers**
  on `[B::Expr; 8]` arrays:
  - `add(a, b)`, `sub(a, b)` тАФ limb-wise sum / difference.
  - `scalar_mul(k_expr, a)` тАФ multiply each limb by a `B::Expr`
    scalar (used for constants and for the Shamir indicator `s`).
  - `mul(a, b)` тАФ schoolbook multi-limb multiplication, **truncated to
    limbs `0..7`**: output `[k] = sum_{i+j=k, i+j<8} a[i] * b[j]`.
  - The 8-limb decompositions of the secp256k1 constants `Gx, Gy, Qx, Qy,
    GQx, GQy, R_SIG` are precomputed at constraint-build time and
    injected via the standard `cst` closure (`from_ref` of a degree-0
    `DensePolynomial<i64, 32>`).

  `s = bтВБ + bтВВ тИТ bтВБ┬╖bтВВ` is built once as a single `B::Expr` (degree 2 in
  the bit columns) and broadcast across the 8 limbs of any expression it
  multiplies. The same is done for `1 тИТ s`, `sel_init`, `sel_final`, and
  the `T_x`, `T_y` table-point limbs (each limb of which is a degree-2
  expression in `(bтВБ, bтВВ)`).

- **Constraints:** **96** total тАФ exactly the 12 logical `EcdsaUair`
  constraints expanded to 8 per-limb assertions each, in the same order:
  `C1, C2, C3, C4, W1 (H_sq), W2 (H_cu), W3 (R_a), C5, C6, C7, B3, B4`.
  The per-limb constraint at position `k` for each logical constraint has
  **the same degree** as the corresponding `EcdsaUair` constraint, since
  the limb cells substitute for `Int<4>` cells one-for-one inside the same
  algebraic shape:

  ```text
  per-constraint degree array =
    [2 ├Ч 8, 4 ├Ч 8, 5 ├Ч 8, 4 ├Ч 8, 2 ├Ч 8, 2 ├Ч 8, 5 ├Ч 8,
     4 ├Ч 8, 4 ├Ч 8, 5 ├Ч 8, 2 ├Ч 8, 4 ├Ч 8]   // 96 entries total
  ```

  **Max degree: 5** тАФ same as current `EcdsaUair`.
- **Trace generator.** Same secp256k1 walk as `EcdsaUair` (reusing
  `point_double`, `point_add_affine`, `select_table_point`, `IDENTITY`,
  `fp_mul`, `fp_sub` from the `ecdsa_secp256k1` helper module). After
  computing each `Uint<4>` value, decompose it into 8 little-endian u32
  limbs via `uint4_to_u32_limbs` and store each as `i64`. `num_vars тЙе 9`
  is required (inherited 258-row minimum).
- **The witness intentionally does NOT satisfy the constraint system** тАФ
  same caveat as `EcdsaUair`. Use only for prover-only benchmarking. See
  `bench_ecdsa_limbs` in [protocol/benches/e2e.rs](../protocol/benches/e2e.rs).
- **Why it exists:** measures prover throughput on a *production-shape*
  multi-limb arithmetization of secp256k1 тАФ the column count is 6├Ч wider
  than `EcdsaUair` (84 vs 14), the constraint count is 8├Ч higher (96 vs
  12), and each per-limb constraint expands to a much larger expression
  tree (cross-limb partial-product sums). The max degree and per-limb
  constraint shape are deliberately preserved so the comparison isolates
  the cost of "many narrow-cell multi-limb constraints" vs. "few
  wide-cell single-shot constraints".

### `ShaProxyEcdsaUair`

A **heterogeneous combined UAIR** that runs both `SHAProxy<Int<4>>` and
`EcdsaUair` on a single 32-column trace, with both constraint sets active
on every row. Designed to stress the prover on a trace whose shape is
representative of stitching multiple sub-circuits together (e.g. SHA-256
+ ECDSA-verify in a real Zinc+ application).

The implementation **reuses both component UAIRs' `constrain_general`
implementations unchanged** by constructing sliced sub-views of the
incoming `up`/`down` `TraceRow`s and forwarding them. This is the same
delegation pattern used by `BigLinearUairWithPublicInput`, extended to
two distinct component UAIRs that operate on disjoint slices of the same
trace row.

- **Layout:** `(14 bp, 0 ap, 18 int)` тАФ 32 columns total.
  - `bp[0..14]` тАФ SHAProxy's 14 binary-poly columns (unchanged).
  - `int[0..14]` тАФ `EcdsaUair`'s 14 int columns, contiguous so the
    EcdsaUair constraint code's hard-coded indices `B1=0 .. H_CU=13`
    work without any re-indexing.
  - `int[14..18]` тАФ SHAProxy's 4 int columns.
- **Public layout:** `(0, 0, 4)` тАФ same 4 public ECDSA int columns
  (`bтВБ`, `bтВВ`, `sel_init`, `sel_final`) at `int[0..4]`.
- **Shifts:** **24** specs total, built by concatenating
  `SHAProxy::<Int<4>>::signature().shifts()` and
  `EcdsaUair::signature().shifts()` and bumping each `source_col` to its
  combined-trace flat index:
  - 17 SHAProxy bp shifts on flat 0..11 (no offset).
  - 4 SHAProxy int shifts on flat 28..31 (originally 14..17, +14 because
    SHAProxy's int cells now sit after 14 ECDSA int cells in the
    combined int group).
  - 3 ECDSA int shifts on flat 18..20 (originally 4..6, +14 because the
    combined trace has a 14-column bp prefix).
  After `UairSignature::new`'s stable sort by `source_col`:
  - `down.binary_poly[0..17]` тАФ SHAProxy's bp shifts.
  - `down.int[0..3]` тАФ ECDSA's `X[t+1]`, `Y[t+1]`, `Z[t+1]` (source_col
    18, 19, 20 тАФ sorted before SHAProxy's int shifts).
  - `down.int[3..7]` тАФ SHAProxy's 4 int shifts (source_col 28..31).
- **Ideal:** `MixedDegreeOneOrXnMinusOne<Int<4>, 32>` тАФ same as
  `SHAProxy<Int<4>>`. EcdsaUair contributes only `assert_zero` calls,
  so its `Self::Ideal = ImpossibleIdeal` is bridged with an
  `unreachable!()`-bodied closure that the framework never calls (verified
  in `degree_counter` and `constraint_counter`).
- **Scalar:** `DensePolynomial<Int<4>, 32>` тАФ matches both component
  UAIRs when SHAProxy is specialized to `R = Int<4>`.
- **Constraints:** **19** total тАФ 12 from EcdsaUair (delegation order:
  ECDSA first, SHAProxy second) followed by 7 from SHAProxy. Per-constraint
  degree array (as asserted in the test):
  ```text
  [ 2, 4, 5, 4, 2, 2, 5, 4, 4, 5, 2, 4,   // ECDSA: C1, C2, C3, C4, W1, W2, W3, C5, C6, C7, B3, B4
    1, 1, 1, 1, 1, 1, 1 ]                  // SHAProxy: 7 linear constraints
  ```
  **Max degree: 5** (inherited from EcdsaUair).
- **Trace generator.** Calls `SHAProxy::<Int<4>>::generate_random_trace`
  and `EcdsaUair::generate_random_trace` with the same `num_vars` and rng,
  then concatenates: 14 SHAProxy bp cols + (14 ECDSA int + 4 SHAProxy int)
  cols. `num_vars тЙе 9` is required (inherited from `EcdsaUair`'s 258-row
  minimum).
- **The witness intentionally does NOT satisfy the constraint system.**
  SHAProxy's sub-witness is satisfying, but the ECDSA sub-witness is not
  (real secp256k1 walk vs. integer-equality constraints тАФ same caveat as
  `EcdsaUair`). Use only for prover-only benchmarking. See `bench_sha_ecdsa`
  in [protocol/benches/e2e.rs](../protocol/benches/e2e.rs).
- **Why it exists:** measures combined prover throughput on a realistic
  multi-circuit trace shape тАФ the most heterogeneous UAIR in the crate
  (binary-poly + int cells side by side, mixed ideals, both linear and
  high-degree non-linear constraints in the same row).
