# UAIRs in `zinc-test-uair`

This crate defines the UAIRs (Univariate AIR-style constraint systems) that
the prover, verifier, PIOP, and end-to-end benchmarks exercise. Each UAIR is
a self-contained example: a column layout, optional row shifts, a constraint
function, and a random-trace generator. The constraints themselves are
written against the `ConstraintBuilder` DSL вҖ” the prover and verifier each
provide their own builder implementations, so the same constraint code is
reused for both witness generation and verification.

This document is a reference for every UAIR currently in
[src/lib.rs](src/lib.rs): what its shape looks like, what it constrains,
what its trace generator produces, and what part of the system it exists to
exercise.

## Conventions

### Column kinds

A `TotalColumnLayout::new(b, a, i)` declares `b + a + i` columns of three
distinct kinds. Each column kind stores a different value type per row:

- **Binary-polynomial columns** (`bp`) hold `BinaryPoly<32>` вҖ” a polynomial
  with 32 coefficients, each restricted to `0` or `1`. Note that the
  *coefficients* are binary; the column entry itself is still a polynomial,
  not a single bit. Useful identities: `bp.eval(1)` is the popcount (number
  of `1` coefficients) and `bp.eval(2)` is the integer encoded by reading
  the coefficients as the bits of a `u32`.
- **Arbitrary-polynomial columns** (`ap`) hold `DensePolynomial<R, 32>` вҖ” a
  polynomial of degree `< 32` with arbitrary coefficients in the host ring
  `R`.
- **Integer columns** (`int`) hold a single `R` element per row (a scalar,
  with no polynomial structure).

Throughout this document, `bp[i]`, `ap[i]`, and `int[i]` refer to the
`i`-th column of each kind.

### Rows: `up` and `down`

The constraint function receives two row views per call:

- `up.X[i]` is the value of column `X[i]` at the current row `t`.
- `down.X[i]` is the value of column `X[i]` at the *shifted* row.

**Important вҖ” `down` is indexed by shift-spec position, not by source
column.** The signature declares a `Vec<ShiftSpec>`; the `j`-th entry of
`down.binary_poly` corresponds to the `j`-th `ShiftSpec` in that vector,
not to `bp[j]`. Concrete examples:

- `BigLinearUair` uses `(0..16).map(|i| ShiftSpec::new(i, 1))`. The shift
  vector is in source-column order, so `down.binary_poly[i]` happens to
  equal `bp[i][t+1]` here.
- `ShaProxy` uses `vec![ShiftSpec::new(0, 1), ShiftSpec::new(4, 4)]`. Only
  columns `0` and `4` are shifted, and `down.binary_poly[0]` is
  `bp[0][t+1]` while `down.binary_poly[1]` is `bp[4][t+4]` вҖ” *not*
  `bp[1][t+1]`. The two shift amounts are also different (1 and 4).

Shifts are zero-padded beyond the trace boundary: the "virtual" row past
the last real row reads as all zeros.

### Constraints

A `ConstraintBuilder` exposes two assertion primitives:

- `b.assert_zero(expr)` вҖ” the expression must reduce to the zero element.
- `b.assert_in_ideal(expr, &ideal)` вҖ” the expression must lie in the given
  ideal of `R[X]`. At verification time, expressions are projected to a
  prime field and the membership check is evaluated there.

Expressions are built from column values using `+`, `-`, `*`, plus two
extras supplied to the constraint function:

- `from_ref(&scalar)` lifts a fixed `Self::Scalar` (a
  `DensePolynomial<R, 32>`) into the expression type вҖ” i.e., a constant
  polynomial term.
- `mbs(&expr, &scalar)` multiplies an existing expression by a fixed
  `Self::Scalar`. This is the only way to multiply a column expression by
  a polynomial constant such as `X` (encoded as `DensePolynomial::new([0, 1])`).

### Ideal types

Two ideal implementations ship today:

- **`RotationIdeal<R, W>`** generated by `X^W - r` for some root `r вҲҲ R`.
  For `W = 1`, an expression `e(X)` lies in `(X - r)` iff `e(r) = 0`
  (i.e., the projected polynomial vanishes at `r` in the field). For
  `W > 1`, membership is checked by reducing `e(X)` modulo `X^W - r`.
  This document writes such ideals as `(X^W - r)`. The roots used in
  practice include `r = 1` (so `(X - 1)` checks "popcount/sum" relations,
  and `(X^w - 1)` checks rotation relations) and `r = 2` (so `(X - 2)`
  checks "integer-decomposition" relations).
- **`ImpossibleIdeal`** has `contains(_) = false`. It is the placeholder
  type used by UAIRs whose constraints are exclusively `assert_zero`
  (no ideal-membership check is ever performed against this type).

## Why every ECDSA UAIR in this crate is non-satisfying

`EcdsaUair`, `EcdsaUairLimbs`, and `ShaProxyEcdsaUair` all encode their
constraints as **exact integer equalities** (every constraint is
`assert_zero` over the cell type) and generate witnesses by performing a
real Shamir's-trick walk over secp256k1 with **modular** F_p arithmetic.
The two are inconsistent on purpose, and the resulting witnesses fail to
satisfy the constraint system. The benches are therefore **prover-only**
вҖ” running the verifier on any of these traces would correctly reject it.

This document records why we never closed that gap, because it looks
like an obvious bug at first glance and will keep looking like one to
future readers. Two distinct issues block a "real walk + satisfying
witness" combination, and **both** would have to be solved together.

### Issue 1: integer arithmetic is not modular arithmetic

The constraint `C1: Z_mid вҲ’ 2В·YВ·Z = 0` says, as written, that the cell
`Z_mid` equals the integer `2В·YВ·Z` exactly. The witness builds `Z_mid`
via `fp_mul`, so `Z_mid = (2В·YВ·Z) mod P` where `P` is the secp256k1
base prime (`вүҲ 2ВІвҒөвҒ¶`). For Y, Z near P, the integer product `2В·YВ·Z` is
close to `2В·PВІ` and the modular reduction is non-trivial: `Z_mid` and
`2В·YВ·Z` differ by some multiple of P, not by zero. The constraint as
written checks integer equality, the witness checks modular equality,
and these are different relations.

The standard fix for this kind of arithmetization gap is to add a
**quotient column** `q` per constraint, rewriting `lhs = 0 (mod P)` as
`lhs вҲ’ qВ·P = 0 (as exact integers)`. The witness sets `q = lhs/P` so
the equation holds exactly, and the prover (which evaluates over a
random PIOP field `F`) sees the integer identity preserved because
projection from в„Ө to `F` is a ring homomorphism. This is well-known and
straightforward in principle. In practice, see Issue 2.

### Issue 2: `Int<4>` projects to `MontyField` with sign-aware semantics

The trace cells are `Int<4>` (256-bit signed integers), and the prover
projects them to its PIOP field `F` (a `MontyField`) via
`FromWithConfig<&Int<N>>`. That impl, in
[`crypto-primitives`](https://github.com/NethermindEth/crypto-primitives.git),
literally does:

```rust
let mut abs = value.inner().abs();
// ... compute MontyField representation of |value| ...
if value.is_negative() { -result } else { result }
```

It takes the **absolute value** of the underlying `Int<4>` and negates
if `is_negative()` (which checks the bit-255 sign bit). This is
mathematically correct for integers but fights us when we want to use
`Int<4>` as a *256-bit container* for canonical F_p representatives in
`[0, P)`.

Concretely: roughly **half** of all canonical F_p elements have bit
255 set (since `P вүҲ 2ВІвҒөвҒ¶ вҲ’ 2ВіВІ вҲ’ 977 > 2ВІвҒөвҒө`). For each such element,
the bit pattern stored in the `Int<4>` cell looks "negative" to
`MontyField`. For instance, the canonical value `P вҲ’ 1 вүҲ 2ВІвҒөвҒ¶ вҲ’ 2ВіВІ вҲ’
978` projects to `MontyField(вҲ’(2ВіВІ + 978))`, **not** to `MontyField(P вҲ’
1)`. These are different field elements (they agree mod `P`, but not
mod the PIOP field's modulus, which is what `MontyField` actually
checks).

Consequence: even if we add quotient columns to fix Issue 1, the
resulting "satisfying" integer identity does not project to a
satisfying `F`-identity, because the sign of about half of the cell
projections is wrong relative to what the witness intended. The
quotient trick produces an integer identity that's only satisfied
under one interpretation of the bit pattern (unsigned в„Ө embedded in
в„Ө), and `MontyField` uses a different one (signed `Int<4>` interpreted
in two's complement and then sign-extended into `F`).

### So why did we ever try?

The first version of `EcdsaUair` was designed before we understood this
projection behavior. At the time, the question seemed to be a simple
choice between "real walk, accept non-satisfying" and "all-zero fixed
point, satisfying but trivial" вҖ” and the user picked the real walk for
trace-shape realism. The plan was always that we could later add
quotient columns to make the real walk satisfying.

That later attempt is what produced this document. Discovering Issue 2
mid-design made it clear that closing the gap requires more than
quotient columns:

- **Switch to limb form** (`i64` cells holding 32-bit u32 limbs, like
  [`EcdsaUairLimbs`](#ecdsauairlimbs)) so that every cell is small and
  positive and the sign-aware projection is unambiguous. Then add
  **quotient + carry columns** to do per-limb modular reduction. This
  is essentially the production approach used by real ECDSA UAIRs in
  other zk libraries; estimated cost is hundreds of new columns and
  ~1000+ lines of constraint code.
- **Switch to `Int<8>` cells** so the canonical 256-bit values fit as
  positive 512-bit signed integers (sign bit is bit 511, never set for
  values in `[0, 2ВІвҒөвҒ¶)`). This solves Issue 2 cleanly with one cell per
  field element, but requires a new `ZincTypes` config and bench
  helper, and pays a per-cell PCS cost penalty.
- **Toy curve over a small prime** (e.g. FвӮҒвӮҖвӮҒ): all field elements
  trivially fit in `i64`, both issues vanish, and the constraint shape
  is preserved. This is the approach taken by v1's `EcdsaUairDp`. The
  downside is that it stops being secp256k1.

Each of these is its own substantial project; none is a small patch on
the existing UAIRs. We chose to leave the current ECDSA UAIRs as
**prover-only benchmark fixtures** with an explicit "non-satisfying
witness" caveat in their per-UAIR sections, rather than fix the gap
in-place.

The prover-only benches (`bench_ecdsa`, `bench_ecdsa_limbs`,
`bench_sha_ecdsa`) still measure realistic prover throughput on
realistic constraint shapes вҖ” they just don't exercise the verifier.
For verifier-running benches we use the satisfying UAIRs in this crate
(`BigLinearUair`, `BigLinearUairWithPublicInput`, `SHAProxy`, etc.).

## Benchmark UAIRs

These UAIRs are sized for the end-to-end criterion benchmark in
[protocol/benches/e2e.rs](../protocol/benches/e2e.rs). They are deliberately
"wide" (many columns and many constraints) so that prover/verifier
performance numbers reflect realistic per-constraint and per-column
overheads rather than fixed-cost setup.

### `BigLinearUair<R>`

A many-column linear UAIR encoding a *popcount-preservation* relation:
each binary-polynomial column is allowed to permute its `1` bits across
rows, but its popcount must stay constant; in addition, column 0 is
forced to be the `BinaryPoly` representation of the row's total popcount
across all 16 columns.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Shifts:** all 16 bp columns shifted by 1 (so `down.binary_poly[i] =
  bp[i][t+1]`, taking advantage of the source-column order in the shift
  vector)
- **Ideal:** `(X - 1)` and `(X - 2)`
- **Constraints (per row `t`):**
  ```
  (bp[0] + bp[1] + вҖҰ + bp[15]) - int[0]            вҲҲ (X - 1)   // int[0] = total popcount
  down.binary_poly[0]          - int[0]            вҲҲ (X - 2)   // bp[0][t+1] encodes that popcount as a u32
  for i in 1..16:
      up.binary_poly[i]        - down.binary_poly[i] вҲҲ (X - 1) // bp[i] preserves popcount across rows
  ```
- **Trace:** row 0 is fully random; for each subsequent row, `bp[1..16]`
  are fresh random binary polynomials that *match the previous row's
  popcount on each column*, and `bp[0][t+1]` is set to
  `BinaryPoly::from(total_popcount)` so that the second constraint holds.
- **Why it exists:** stresses the prover's per-column and per-constraint
  cost (16 columns Г— 17 constraints, all degree 1) without introducing any
  non-linearity. The matching MLE-first prover path also runs because the
  max constraint degree is 1.

### `BigLinearUairWithPublicInput<R>`

Identical to `BigLinearUair` (same layout, shifts, constraints, trace),
but the first 4 binary-polynomial columns are marked as public input.
Exists so the end-to-end benchmark can also measure the public-input
plumbing on the same constraint shape.

- **Layout:** `(16 bp, 0 ap, 1 int)`
- **Public layout:** `(4 bp, 0 ap, 0 int)` вҖ” first 4 bp columns are public
- **Shifts / ideal / constraints / trace:** delegated to `BigLinearUair`

### `ShaProxy<R>`

A second large linear UAIR with a different shape from `BigLinearUair`:
14 binary-polynomial columns, 4 integer columns, only 2 shifted columns
(with *different* shift amounts: 1 and 4), and constraints that mix
binary-polynomial and integer summations plus a multiply-by-`X` relation.
The name reflects the inspiration вҖ” it has a similar high-level shape to
bit/integer constraints found in proxies for SHA-style hash circuits вҖ”
even though it does not actually compute a SHA round.

- **Layout:** `(14 bp, 0 ap, 4 int)`
- **Shifts:** `bp[0]` shifted by 1, `bp[4]` shifted by 4
  - so `down.binary_poly[0] = bp[0][t+1]` and
    `down.binary_poly[1] = bp[4][t+4]` (note both the index gap *and*
    the asymmetric shift amounts вҖ” see the Conventions section's
    warning)
- **Ideal:** `(X - 1)` and `(X - 2)`
- **Constraints (per row `t`; let `dbp_c1 = bp[0][t+1]` and `dbp_c5 = bp[4][t+4]`):**
  ```
  C1:  dbp_c1 - bp[1] - bp[2] - bp[3] - int[0] - int[1] - int[2]   вҲҲ (X - 2)
  C2:  dbp_c5 - bp[5] - bp[6] - bp[7] - int[1] - int[2] - int[3]   вҲҲ (X - 2)
  C3:  bp[8]  - int[0]                                              вҲҲ (X - 2)
  C4:  bp[9]  - int[1]                                              вҲҲ (X - 2)
  C5:  bp[10] - X * bp[11]                                          вҲҲ (X - 1)
  C6:  bp[12] - X * bp[13]                                          вҲҲ (X - 1)
  ```
  In C5/C6, `X *` is encoded via
  `mbs(&col, &DensePolynomial::new([0, 1]))`.
- **Trace:** witnesses are constructed so each constraint holds *exactly*:
  - **C5/C6.** Pick `bp[11]` and `bp[13]` uniformly at random over all
    `BinaryPoly<32>` values. Then choose `bp[10]` (resp. `bp[12]`) to be
    a random binary polynomial with the *same popcount* as `bp[11]`
    (resp. `bp[13]`). At `X = 1` we have `(X * bp[k]).eval(1) =
    bp[k].eval(1) = popcount(bp[k])`, so `bp[10] - X*bp[11]` vanishes at
    `X = 1` whenever the popcounts match.
  - **C3/C4.** Pick small random `u32` values for `int[0]` and `int[1]`
    (uniformly in `0..32`), then set `bp[8] := BinaryPoly::from(int[0])`
    and `bp[9] := BinaryPoly::from(int[1])`. Because
    `BinaryPoly::from(n).eval(2) == n` for `n < 2^32`, the difference
    vanishes at `X = 2` exactly.
  - **C1/C2.** The summands `bp[1..=3]` and `bp[5..=7]` are random binary
    polynomials capped at 28 coefficient bits (so each `eval(2) вүӨ 2^28 вҲ’
    1`); `int[0..=3]` are small `u32`s in `0..32`. The cap guarantees
    that the row-`t` sums (e.g.
    `SвӮҒ = bp[1].eval(2) + bp[2].eval(2) + bp[3].eval(2) + int[0] + int[1] + int[2]`)
    fit in a `u32` (worst case `вүҲ 8 Г— 10вҒё`, well below `2ВіВІ вҲ’ 1`), so
    each constraint can be discharged by setting
    `bp[0][t+1] := BinaryPoly::from(SвӮҒ as u32)` (and analogously
    `bp[4][t+4] := BinaryPoly::from(SвӮӮ as u32)`).
  - **C2 boundary handling.** Because C2's shift is 4, the rows
    `i вҲҲ {lenвҲ’4, lenвҲ’3, lenвҲ’2}` would otherwise read `bp[4][i+4]` past
    the trace boundary as zero-padding. To still satisfy C2 at those
    rows, the trace generator zeros out `bp[5..=7]` and `int[1..=3]` on
    those three rows so the C2 RHS sum becomes `0 вҲ’ 0 = 0` and the
    constraint vanishes trivially. (Row `len вҲ’ 1` is exempt from
    constraint checking by the protocol's last-row selector and so does
    not need any special handling.) This mirrors the boundary trick used
    by `TestUairMixedShifts` for its `c[i] = b[i+2]` relation.
  - **`bp[4]` head rows.** Symmetrically, since `bp[4]` is shifted by 4,
    the C2 fix-up at iteration `i` writes `bp[4][i+4]` вҖ” so
    `bp[4][0..4]` are never written by the loop. The generator
    initializes those four head rows to small random binary polynomials
    up front; their values are unconstrained because the protocol does
    not check virtual rows before the start of the trace either.

> **Note on `(X^32 - 1)`:** the original specification of `ShaProxy`
> wrote C5/C6 against the cyclotomic ideal `(X^32 - 1)`, where multiplying
> by `X` corresponds to a cyclic shift of coefficients. The codebase
> currently only uses `RotationIdeal<_, 1>`, so we substituted `(X - 1)`.
> The new constraint is *strictly weaker* вҖ” it only checks popcount
> equality between the paired columns instead of full cyclic-shift
> equality вҖ” but the constraint *shape* (a degree-1 ideal check on a
> two-term expression involving a multiply-by-`X`) is preserved, which is
> what determines prover/verifier work for the benchmark.

## Quick reference

Column counts are listed as `BinPoly / ArbPoly / Int`, matching the
`(binary_poly, arbitrary_poly, int)` ordering used by
`TotalColumnLayout::new`. "Shifts" are written as `count Г— amount`.
