% ===========================================================================
\section{Arithmetizing the ECDSA building blocks}\label{sec:arith}
% ===========================================================================

We collect the arithmetic identities that translate each step of the
double-and-add loop into a constraint over $\F_p$. Throughout this
section, all arithmetic is in $\F_p$ unless noted otherwise; we
suppress reductions $\bmod\,p$.

\subsection{Shamir's trick}\label{sec:shamir}

The standard binary double-and-add algorithm computes a single
scalar multiplication $u A$ (for a $k$-bit scalar $u$ and a point $A$)
by initializing an accumulator $P \leftarrow \OOO$ and, for
$j = k-1, \dots, 0$, performing
$$
P \;\leftarrow\; 2P, \qquad
P \;\leftarrow\; P + A \quad \text{whenever bit $j$ of $u$ is $1$.}
$$
Naively, computing the two-term sum $u_1 G + u_2 Q$ would run two
such loops for a total of $2k$ doublings and up to $2k$ additions.
Shamir's trick merges them into a single loop. Precompute the four
points
$$
T_{00} = \OOO,\quad T_{10} = G,\quad T_{01} = Q,\quad T_{11} = G + Q,
$$
initialize $P \leftarrow \OOO$, and run, for $j = k-1, \dots, 0$,
$$
P \;\leftarrow\; 2P, \qquad
P \;\leftarrow\; P + T_{b_1[j]\,b_2[j]}.
$$
After $k = 256$ steps, $P = u_1 G + u_2 Q$. The total cost is $k$
doublings and at most $k$ mixed additions.

In the UAIR, the per-row affine addend
$T_{b_1[t]\,b_2[t]} \in \{\OOO, G, Q, G+Q\}$ is supplied by the
verifier in two public columns $\col{PA\_X\_ADDEND}$ and
$\col{PA\_Y\_ADDEND}$, with a separate boolean public selector
$\col{S\_ADD}$ flagging whether the addend is non-identity (i.e.\
$\col{S\_ADD}[t] = 1 \iff (b_1[t], b_2[t]) \neq (0, 0)$). When
$\col{S\_ADD}[t] = 0$ the addition step at row $t$ collapses to
$P \leftarrow 2P$.

\begin{remark}[Verifier-supplied addend]\label{rem:addend}
$Q$ and $G+Q$ are public (\Cref{rem:public-u}), so the four-point
precompute $\{\OOO, G, Q, G+Q\}$ is public, and the per-row addend
choice is determined by the public bits $(b_1[t], b_2[t])$. The
implementation has the verifier write the chosen affine addend
directly into $(\col{PA\_X\_ADDEND}, \col{PA\_Y\_ADDEND})$ for each
row, rather than committing four selector polynomials and adding
in-circuit constraints to compute the addend from the bits. The
cost is that the verifier must do this $256$-row table lookup
itself before committing to the public input.
\end{remark}

\begin{remark}[Cost of in-circuit addend selection]\label{rem:addend-cost}
The alternative — having the prover (or the constraints) compute the
addend from $(b_1[t], b_2[t])$ in-circuit — incurs the following
cost relative to the deployed design:
\begin{itemize}[leftmargin=2em]
  \item \textbf{Two extra public selector columns} carrying the
    bit pair $(b_1, b_2)$ row-wise (each in $\{0,1\}$); each takes
    one $\F_p$-typed integer column under the
    storage convention of \Cref{sec:cols}, so $+2$ public columns.
    These columns must be range-pinned to $\{0,1\}$ —
    either via two booleanity-style ideal constraints
    $b_i (b_i - 1) \in (p)$ ($+2$ degree-$2$ constraints) or via a
    direct verifier inspection of \texttt{public\_trace} (per the
    SHA UAIR's verifier-side public-column structural-check
    pattern).
  \item \textbf{Two new addend-selection constraints,
    $T_X$ and $T_Y$.} Using
    $T = (1-b_1)(1-b_2) \OOO + b_1 (1-b_2) G + (1-b_1) b_2 Q + b_1 b_2 (G+Q)$
    coefficient-wise, each coordinate is a degree-$2$
    polynomial in $(b_1, b_2)$ with publicly-known $\F_p$ scalars,
    pinned in $(p)$. With $T_X, T_Y$ then \emph{also} entering the
    mixed-addition formulas $H = T_X P_{\mathrm{pa},Z}^2 - P_{\mathrm{pa},X}$
    and $R_a = T_Y P_{\mathrm{pa},Z}^3 - P_{\mathrm{pa},Y}$ as
    full degree-$2$ expressions in $(b_1, b_2)$, the
    \emph{effective constraint degree} of \eqref{eq:H},
    \eqref{eq:Ra} (and hence of every downstream addition
    constraint that consumes $H, R_a$) rises from $\le 4$ to
    $\le 6$.
  \item \textbf{Booleanity for $\col{S\_ADD}$.} The
    non-identity selector $\col{S\_ADD}[t] = b_1[t] + b_2[t] - b_1[t] b_2[t]$
    is also a degree-$2$ public polynomial in the bits.
\end{itemize}
The deployed design absorbs all of the above into one
verifier-side $256$-row table lookup, paying instead two precomputed
$\F_p$-valued public columns $(\col{PA\_X\_ADDEND}, \col{PA\_Y\_ADDEND})$.
Net savings: $0$ added witness columns (vs.\ $2$); no constraint-degree
elevation (the deployed UAIR's effective degree stays at the value
set by the doubling/addition formulas alone); no extra ideal
constraints. The verifier's added work is $4 \cdot 256 = 1024$
$\F_p$ multiplications/additions to materialise the addend column —
trivially below mp\_eval-side costs.
\end{remark}

\subsection{Jacobian doubling}\label{sec:doubling}

Given an input Jacobian point $\jac{P}$, the Jacobian doubling
$2P = (P_{\mathrm{pa},X} : P_{\mathrm{pa},Y} : P_{\mathrm{pa},Z})$ is
computed in $\F_p$ by
\begin{align}
P_{\mathrm{pa},Z} &= 2 P_Y P_Z, \label{eq:dbl-z}\\
P_{\mathrm{pa},X} &= 9 P_X^4 - 8 P_X P_Y^2, \label{eq:dbl-x}\\
P_{\mathrm{pa},Y} &= 12 P_X^3 P_Y^2 - 3 P_X^2 P_{\mathrm{pa},X} - 8 P_Y^4. \label{eq:dbl-y}
\end{align}
These are the standard short-Weierstrass $a = 0$ doubling formulas.
The intermediate $S = P_Y^2$ is computed inline (the implementation
inlines it rather than committing it as a separate column).

\subsection{Mixed Jacobian-affine addition}\label{sec:add}

Given $(P_{\mathrm{pa},X} : P_{\mathrm{pa},Y} : P_{\mathrm{pa},Z})$
(the doubled point) and an affine addend $(T_X, T_Y)$ from the
precomputed $\{G, Q, G+Q\}$, the mixed addition
$P_{\mathrm{pa}} + T = (P_{\mathrm{add},X} : P_{\mathrm{add},Y} : P_{\mathrm{add},Z})$
proceeds via the scratch
quantities
\begin{align}
H   &:= T_X \cdot P_{\mathrm{pa},Z}^2 \;-\; P_{\mathrm{pa},X}, \label{eq:H}\\
R_a &:= T_Y \cdot P_{\mathrm{pa},Z}^3 \;-\; P_{\mathrm{pa},Y}, \label{eq:Ra}
\end{align}
followed by the addition outputs
\begin{align}
P_{\mathrm{add},Z} &= P_{\mathrm{pa},Z} \cdot H, \label{eq:add-z}\\
P_{\mathrm{add},X} &= R_a^2 - H^3 - 2 P_{\mathrm{pa},X} H^2, \label{eq:add-x}\\
P_{\mathrm{add},Y} &= R_a \bigl(P_{\mathrm{pa},X} H^2 - P_{\mathrm{add},X}\bigr) - P_{\mathrm{pa},Y} H^3. \label{eq:add-y}
\end{align}
The implementation commits only $H$ (named $\col{W\_C}$) and $R_a$
(named $\col{W\_D}$), inlining $P_{\mathrm{pa},Z}^2$, $P_{\mathrm{pa},Z}^3$,
$H^2$, $H^3$, and $P_{\mathrm{pa},X} H^2$ into the surrounding
constraint expressions. This trades higher per-constraint algebraic
degree for fewer committed columns; with the inlined form the maximum
constraint degree is~$6$ (attained in \eqref{eq:add-y} after
selector gating, see \Cref{sec:constraints}).

\subsection{Conditional addition via output-selection}\label{sec:cond-add}

Rather than gating the addition step with a multiplicative selector
that would inflate the constraint degree, the implementation folds
the conditional add into the row-chaining itself. Let
$\col{S\_ADD}[t] \in \{0, 1\}$ be the per-row addition selector
(\Cref{rem:addend}). The next-row Jacobian state is then defined as
\begin{align}
P_{\mathrm{next},X} &= P_{\mathrm{pa},X} + \col{S\_ADD}\bigl(P_{\mathrm{add},X} - P_{\mathrm{pa},X}\bigr), \label{eq:next-x}\\
P_{\mathrm{next},Y} &= P_{\mathrm{pa},Y} + \col{S\_ADD}\bigl(P_{\mathrm{add},Y} - P_{\mathrm{pa},Y}\bigr), \label{eq:next-y}\\
P_{\mathrm{next},Z} &= P_{\mathrm{pa},Z} + \col{S\_ADD}\bigl(P_{\mathrm{add},Z} - P_{\mathrm{pa},Z}\bigr), \label{eq:next-z}
\end{align}
so that $\col{S\_ADD} = 1$ selects $P_{\mathrm{add}}$ and
$\col{S\_ADD} = 0$ selects $P_{\mathrm{pa}}$ (the pure doubling result).

Because $\col{S\_ADD}$ is a public boolean, $\col{S\_ADD} (1 - \col{S\_ADD}) = 0$
need not be enforced in-circuit; the verifier supplies the column with
$0/1$ values directly.

\begin{remark}[Inlined addition collapses to a single output-selection constraint]
\label{rem:inlined-add}
Substituting $P_{\mathrm{add}, X/Y/Z}$ from
\eqref{eq:add-z}--\eqref{eq:add-y} into \eqref{eq:next-x}--\eqref{eq:next-z}
and inlining all of $P_{\mathrm{pa},Z}^2, H^2, H^3, P_{\mathrm{pa},X} H^2$
yields three constraints whose maximum algebraic degree (in committed
columns) is~$6$. The deg-$6$ monomial in the $Y$ output-selection
constraint is exactly $\col{S\_ACTIVE} \cdot \col{S\_ADD} \cdot D \cdot X_{\mathrm{pa}} \cdot C^2$,
matching the spec's $s_{\mathrm{reg}} \cdot R_a \cdot X_{\mathrm{mid}} \cdot H^2$
term.
\end{remark}

\subsection{Initial step and final state}\label{sec:boundary}

\paragraph{Init boundary.}
At row~$0$ the Jacobian state is pinned to the verifier-supplied
non-identity seed $R_{\mathrm{init}} = (R_{\mathrm{init},X} : R_{\mathrm{init},Y} : R_{\mathrm{init},Z})$
(\Cref{rem:r-init}):
$$
\bigl(P_X[0], P_Y[0], P_Z[0]\bigr) \;=\; \bigl(\col{PA\_R\_INIT\_X}, \col{PA\_R\_INIT\_Y}, \col{PA\_R\_INIT\_Z}\bigr).
$$
This is enforced by three zero-ideal constraints gated by
$\col{S\_INIT}$ (see \Cref{con:init-x}--\Cref{con:init-z}).

\paragraph{Final state and off-protocol affine readout.}
At the final row $t = \mathrm{FINAL\_ROW} := \mathrm{NUM\_SHAMIR\_ROUNDS} = 256$,
the witness Jacobian state $\jac{P}[256]$ holds the result
$R = u_1 G + u_2 Q$ (modulo the $R_{\mathrm{init}}$ offset of
\Cref{rem:r-init}). In the deployed UAIR, the affine $x$-coordinate
$R_x = P_X[256] \cdot P_Z[256]^{-2} \bmod p$ is \emph{not} extracted
in-circuit, and the signature-defining identity
$R_x \equiv r \pmod n$ is \emph{not} enforced. The intent is that
the verifier opens the three witness cells at row $256$, performs a
single field inversion, and checks the conditions off-protocol; the
selector column $\col{S\_FINAL}$ is retained in the layout for a
downstream UAIR that wants to add the missing final-row constraints.

\begin{remark}[The final affine-readout constraint is missing]\label{rem:final-readout}
The above design is, strictly speaking, \emph{incomplete} as an
arithmetization of ECDSA verification: a sound ECDSA proof must
bind the in-circuit Jacobian computation to the signature scalar
$r$ via $R_x \equiv r \pmod n$. The deployed UAIR proves only that
the witness Jacobian state at row $256$ is \emph{some} point of
the form $u_1 G + u_2 Q$ (with $u_1, u_2$ supplied as public
input); it does \emph{not} prove that the affine $x$-coordinate of
that point matches the signature's $r$. A complete UAIR therefore
needs:
\begin{itemize}[leftmargin=2em]
  \item One witness column $Z_{\mathrm{inv}}$ (or a single cell
    pinned at row $256$) carrying $P_Z[256]^{-1} \bmod p$, plus the
    in-circuit non-infinity check
    \[
      S_{\mathrm{FINAL}} \cdot \bigl(P_Z[256] \cdot Z_{\mathrm{inv}}[256] - 1\bigr) \in (p).
    \]
  \item One witness column $R_x$ (or a final-row cell) plus the
    affine-derivation constraint
    \[
      S_{\mathrm{FINAL}} \cdot \bigl(P_X[256] \cdot Z_{\mathrm{inv}}[256]^2 - R_x[256]\bigr) \in (p).
    \]
  \item A constraint binding $R_x \equiv r \pmod n$, where $r$
    enters as a public column. This step takes place in the order
    field $\F_n$, which is distinct from the base field $\F_p$ used
    everywhere else, so it requires either:
    \begin{itemize}
      \item adding $n$ to the index $\mathbf{q}$, i.e.\
        $\mathbf{q} = (p, n)$, with the matching $(n)$-ideal
        constraint $S_{\mathrm{FINAL}} \cdot (R_x[256] - r) \in (n)$;
        \emph{or}
      \item proving $R_x[256] \in [0, n)$ via a range argument and
        then checking equality directly. Because $p_\secp$ and
        $n_\secp$ are extremely close (they differ in the bottom
        bits only), the typical $R_x$ value lies in $[0, n)$ already
        and the range-check has effective probability $\approx 1$.
    \end{itemize}
\end{itemize}
The first route adds one $\F_n$-branch column plus one $(n)$-ideal
constraint. The second route adds one bit-decomposition of $R_x$
plus a sub-$n$ comparison.

The deployed UAIR omits these for two reasons: \emph{(i)}~the
implementation is targeted at composition (the $\col{S\_FINAL}$
selector is reserved precisely so a downstream UAIR can attach the
final-row constraints without disturbing the present layout), and
\emph{(ii)}~adding the $\F_n$ branch turns the index from a
single-prime $\mathbf{q} = (p)$ into the multi-prime
$\mathbf{q} = (p, n)$, which exercises code paths beyond the scope
of this report. The final-row affine readout is therefore left
out-of-band; this report should be read as describing the
\emph{Jacobian-loop sub-arithmetization} of ECDSA, not a complete
end-to-end ECDSA verification UAIR.
\end{remark}
